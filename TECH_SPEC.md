# Техническое задание (ТЗ)
## Проект: Десктопная система учёта продаж и поступлений для церковной лавки (Linux, ultra-light)

**Версия ТЗ:** 1.0  
**Дата:** 2026-03-27  
**Целевая ОС:** Ubuntu LTS (в т.ч. Xfce/LXQt/Openbox-сборки)  
**Язык реализации:** Rust

---

## 1. Цель и назначение
Разработать локальное десктопное приложение для автоматизации учёта:
- продаж товаров в церковной лавке;
- поступлений товаров на склад;
- формирования отчётности (в т.ч. оборотно-сальдовая ведомость за период);
- ролевого разграничения доступа;
- работы с QR-кодами сотрудников и товаров;
- резервного копирования БД на сетевой ресурс по SFTP/SCP.

Ключевое требование: **работа на слабых ПК** (минимальная нагрузка на CPU/RAM/диск), включая ноутбуки с ограниченными ресурсами.

---

## 2. Область применения
Система предназначена для малого и среднего церковного прихода с одной или несколькими торговыми точками, где требуется:
- прозрачный и проверяемый учёт движения товаров;
- простая работа кассира/кладовщика;
- контроль закрытия дня;
- регламентированная отчётность для ответственных лиц.

---

## 3. Термины и сокращения
- **Лавка** — модуль регистрации продаж.
- **Склад** — модуль регистрации поступлений и остатка товара.
- **ОСВ** — оборотно-сальдовая ведомость.
- **Роль** — набор прав доступа пользователя.
- **Закрытие дня** — фиксация операционного дня, после которой операции этого дня ограничены/запрещены к изменению.
- **QR-сканер** — проводной USB-сканер, работающий в режиме HID-клавиатуры (keyboard wedge).

---

## 4. Общие требования к решению

### 4.1. Техническая платформа
1. Приложение реализуется на **Rust stable**.
2. Приложение работает полностью **локально**, без обязательной зависимости от интернета.
3. Поддержка Linux-дистрибутивов на базе Ubuntu (приоритет: Light Ubuntu/Xfce/LXQt).
4. Архитектура: монолитное локальное приложение (GUI + бизнес-логика + локальная БД).

### 4.2. Требования к лёгковесности
1. Время старта приложения: не более 3 секунд на целевом минимальном ПК.
2. Потребление RAM в простое: целевой диапазон 80–180 МБ.
3. Отказ от тяжёлых web-runtime (браузерных движков) в базовой сборке.
4. Использование нативного/лёгкого GUI-фреймворка для Rust.

### 4.3. Рекомендованный стек (базовый)
- **GUI:** `egui/eframe` (лёгкий immediate mode) или `iced` (при подтверждённой производительности).
- **Локальная БД:** `SQLite` (файл `.db`) с режимом WAL.
- **ORM/доступ к БД:** `sqlx` или `rusqlite`.
- **Миграции:** встроенный механизм миграций при старте.
- **Логирование:** `tracing` + ротация логов.
- **Криптография паролей:** `argon2id`.
- **Резервное копирование:** запуск `sftp/scp` либо встроенный SFTP-клиент (например, на базе `ssh2`).

Допускается файловая БД (SQLite) как основной вариант, поскольку она минимальна по ресурсам и проста в сопровождении.

## 5. Функциональные требования

## 5.1. Авторизация и аутентификация
Система должна поддерживать 2 способа входа:
1. **Логин + пароль**.
2. **Персональный QR-код сотрудника** (сканер вводит строковый токен, система распознаёт и авторизует пользователя).

Дополнительно:
- Ограничение количества неуспешных попыток входа (настраиваемо).
- Блокировка пользователя после N неверных попыток (с разблокировкой администратором).
- Хранение только хеша пароля (argon2id), без хранения паролей в открытом виде.

## 5.2. Модуль «Лавка»
Назначение: регистрация продаж.

Функции:
1. Создание документа продажи (дата/время, сотрудник, список товаров, количество, цена).
2. Добавление товара:
   - сканированием **индивидуального QR-кода товара**;
   - выбором из справочника (поиск по названию/артикулу).
3. Автоматический расчёт итоговой суммы документа.
4. Проверка наличия товара на остатке.
5. Проведение продажи с уменьшением складского остатка.
6. Печать/экспорт чека (опционально, если подключён принтер).
7. Просмотр и фильтрация продаж за период.

## 5.3. Модуль «Склад»
Назначение: регистрация поступлений и управление остатками.

Функции:
1. Создание документа поступления (поставщик/источник, дата, список товаров, количество, закупочная/учётная цена).
2. Добавление новых номенклатурных позиций.
3. Оприходование товара с увеличением остатка.
4. Просмотр текущих остатков.
5. Журнал движений товара (приход/расход/корректировка).

## 5.4. Модуль «Отчёты»
Назначение: построение и запуск отчётов.

Обязательные отчёты:
1. Оборотно-сальдовая ведомость за месяц.
2. ОСВ за произвольный период.
3. Продажи по товарам за период.
4. Движение товара по складу за период.
5. Остатки на дату.

Требования:
- Формирование отчётов в таблице на экране.
- Экспорт в CSV (обязательно), XLSX/PDF (желательно).
- Сохранение параметров «готовых» отчётов (шаблоны отчётов).

## 5.5. Модуль «Администрирование»
Назначение: управление пользователями, ролями, правами и параметрами системы.

Функции:
1. Создание/редактирование/блокировка пользователей.
2. Назначение ролей.
3. Генерация и привязка персонального QR-кода сотрудника.
4. Управление правами доступа к разделам и операциям.
5. Настройка параметров резервного копирования.

## 5.6. Закрытие операционного дня
1. Доступно пользователю с ролью **Супервизор** (и Администратор).
2. После закрытия дня:
   - запрещается изменение/удаление документов закрытого дня (или только по специальному праву «перераскрытие дня»);
   - отчёты по закрытому периоду считаются фиксированными.
3. Фиксируется журнал события: кто, когда, какой день закрыл.

---

## 6. Ролевая модель доступа

## 6.1. Перечень ролей
1. **Работник лавки**
   - видит только раздел «Лавка»;
   - права: чтение/запись в «Лавке».

2. **Работник склада**
   - видит разделы «Лавка» и «Склад»;
   - «Лавка»: только чтение;
   - «Склад»: чтение/запись.

3. **Супервизор**
   - видит «Лавка» и «Склад»;
   - чтение/запись в обоих;
   - может выполнять закрытие дня.

4. **Администратор**
   - видит все разделы: «Лавка», «Склад», «Отчёты», «Администрирование»;
   - полный доступ;
   - управление пользователями, ролями, правами, резервным копированием.

## 6.2. Правила отображения интерфейса
Левое меню (разделы) отображается **динамически** в зависимости от ролей текущего пользователя.

---

## 7. Требования к графическому интерфейсу

## 7.1. Общая компоновка
1. **Левая вертикальная панель разделов** (навигация):
   - Лавка
   - Склад
   - Отчёты
   - Администрирование
2. **Основная рабочая область** справа:
   - табличные формы (гриды);
   - фильтры (дата, сотрудник, товар, статус);
   - кнопки операций (создать, провести, экспортировать, закрыть день).

## 7.2. UX-принципы для слабых ПК
1. Минимум анимаций и визуальных эффектов.
2. Пагинация таблиц и/или виртуальный скролл.
3. Ленивые выборки данных из БД.
4. Горячие клавиши для быстрых операций.
5. Поддержка сканера QR как клавиатурного ввода в активное поле.

## 7.3. Интерфейс авторизации
1. Вкладка/переключатель «Логин/пароль» и «QR-вход».
2. Индикатор успешного/ошибочного сканирования QR.
3. Таймаут сессии при неактивности (настраиваемый).

---

## 8. Требования к учёту товаров и данным

## 8.1. Обязательные сущности
- Пользователь
- Роль
- Право
- Сотрудник QR-идентификатор
- Товар
- QR-код товара
- Документ поступления
- Документ продажи
- Строка документа
- Операционный день
- Событие закрытия дня
- Настройки резервного копирования
- Журнал аудита

## 8.2. Минимальные атрибуты товара
- ID
- Наименование
- Артикул/код
- Единица измерения
- Текущий остаток
- Учётная цена
- Статус (активен/архив)
- QR-код

## 8.3. Журналирование и аудит
Система должна фиксировать:
- входы/выходы пользователей;
- создание/изменение/удаление документов;
- закрытие/переоткрытие дня;
- ошибки резервного копирования.

---

## 9. Требования к QR-сканированию
1. Поддержка проводных USB-сканеров QR, работающих как HID-клавиатура.
2. Возможность настройки префикса/суффикса сканера (если устройство добавляет Enter/Tab).
3. Для сотрудников: сканируемый QR должен содержать уникальный идентификатор сотрудника (или подписанный токен).
4. Для товаров: сканируемый QR должен содержать уникальный код товара.
5. Обработка ошибок:
   - неизвестный QR;
   - неактивный сотрудник;
   - архивный товар;
   - отсутствие прав.

---

## 10. Резервное копирование и восстановление

## 10.1. Общие требования
1. Резервируется весь файл БД SQLite + (опционально) вложения/настройки.
2. Поддержка ручного и автоматического расписания backup.
3. Точка назначения — удалённый ресурс по **SCP/SFTP**.

## 10.2. Настраиваемые параметры
- Хост, порт, логин.
- Путь на удалённом сервере.
- Тип транспорта: SFTP или SCP.
- Способ авторизации: пароль/SSH-ключ.
- Расписание (ежедневно, еженедельно, по времени).
- Ротация (хранить N последних копий).

## 10.3. Безопасность backup
- Шифрование канала (SSH).
- Маскирование чувствительных параметров в логах.
- Проверка успешности передачи (размер/хеш/код завершения).
- Отдельный журнал задач резервного копирования.

## 10.4. Восстановление
- Возможность восстановить БД из выбранной резервной копии.
- Перед восстановлением — создание локальной safety-копии текущей БД.

## 11. Нефункциональные требования

## 11.1. Производительность
1. Операция сканирования и добавления товара в документ: до 300 мс в типовом сценарии.
2. Открытие списка продаж/поступлений за месяц: до 2 сек при объёме до 100k строк.
3. Формирование ОСВ за месяц: до 5 сек при объёме до 1 млн движений (с индексами).

## 11.2. Надёжность
1. Устойчивость к отключению питания:
   - использование транзакций БД;
   - WAL-режим SQLite.
2. Целостность данных при конкурентном доступе в рамках одной машины.

## 11.3. Безопасность
1. Пароли — только в виде хеша (argon2id + соль).
2. Разграничение прав по ролям и операциям.
3. Журнал аудита без возможности обычного удаления пользователем.
4. Защита от SQL-инъекций (параметризованные запросы).

## 11.4. Сопровождаемость
1. Логи ошибок и действий.
2. Конфиг в отдельном файле (`.toml`/`.yaml`) с валидацией.
3. Автоматические миграции схемы БД.

## 13. Требования к развёртыванию
1. Формат поставки: `.deb` пакет и/или AppImage.
2. Установка без сложных внешних зависимостей.
3. Первый запуск: мастер первичной настройки (админ, путь БД, backup-параметры).

---

## 14. Ограничения и допущения
1. Базовый режим — **однопользовательский/локальный на одной машине**.
2. Работа нескольких пользователей одновременно на разных машинах требует отдельного этапа (клиент-серверная архитектура).
3. Для QR-сканера предполагается поддержка режима HID без установки проприетарных драйверов.

## 15. Критерии приёмки
Система считается принятой при выполнении всех условий:
1. Реализованы модули: Лавка, Склад, Отчёты, Администрирование.
2. Реализована ролевая модель с корректной видимостью разделов.
3. Работает вход по логину/паролю и по QR сотрудника.
4. Работает добавление товара в продажу по QR товара.
5. Реализовано закрытие дня с фиксацией в журнале.
6. Формируются обязательные отчёты (включая ОСВ за месяц и за произвольный период).
7. Реализован backup файла БД на удалённый ресурс по SCP/SFTP.
8. Приложение корректно работает на целевой лёгкой Linux-среде и не превышает установленные лимиты производительности в типовых сценариях.
